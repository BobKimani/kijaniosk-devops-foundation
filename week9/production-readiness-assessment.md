# Production Readiness Assessment

The current Kubernetes deployment is production-approaching, but it is not yet ready to serve real KijaniKiosk customers. The Ingress setup proves that kk-api and kk-payments can be routed externally through a single host and path-based rules, but the current configuration is still basic. A production Ingress should include TLS termination using cert-manager and a ClusterIssuer, with a tls section added to the Ingress manifest. The NGINX Ingress should also include rate limiting annotations such as nginx.ingress.kubernetes.io/limit-rps and nginx.ingress.kubernetes.io/limit-connections to reduce abuse or accidental traffic spikes. For a payment-facing system, authentication and authorization should be enforced at the application layer, and where appropriate, the Ingress could integrate external authentication using nginx.ingress.kubernetes.io/auth-url. Custom error handling would also be needed so customers do not see raw backend or gateway errors.

The current readiness and liveness probes are useful, but they would need refinement for a real payment service. The readiness probe on /health confirms that traffic should only reach Pods that are ready, but in production I would separate /ready from /live. The /ready endpoint should check critical dependencies such as database connectivity, payment provider configuration, and required environment variables. The /live endpoint should only confirm that the process is alive. I would also tune initialDelaySeconds, periodSeconds, and failureThreshold using observed kk-payments startup behaviour. If startup is slow, a higher initialDelaySeconds or a startupProbe would prevent Kubernetes from restarting healthy but slow-starting containers too early.

Three kk-payments replicas are a good baseline, but manual scaling is not enough for end-of-month traffic spikes. Autoscaling would require metrics-server, accurate CPU and memory requests, and a HorizontalPodAutoscaler for kk-payments. A starting HPA could scale from 3 to 10 replicas based on CPU utilization, but request rate or latency metrics would be better for payment traffic. Setting the HPA target too low could over-scale and waste resources, while setting it too high could delay scaling until users already experience slow responses. Production would also need monitoring, alerts, centralized logs, PodDisruptionBudgets, NetworkPolicies, and a safer secret management approach such as External Secrets, Sealed Secrets, or a cloud secret manager.
