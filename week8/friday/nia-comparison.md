# Week 8 Delivery Pipeline Comparison for Nia

This week’s delivery approach gives KijaniKiosk a more repeatable and safer way to move the payments service from source code to a running system. In Week 7, the service could be switched between two prepared environments, which gave the team a useful rollback path. However, much of the confidence still depended on machine setup, local scripts, and careful human execution. Week 8 changes the focus from switching between prepared folders to delivering a packaged application that can be rebuilt, stored, pulled, started, replaced, and verified in a consistent way.

The strongest improvement is repeatability. The payments service is now packaged into a smaller production image that contains only what is needed to run the service. The baseline image was 556MB, while the production image is 96.1MB. That is a reduction of 459.9MB, or about 82.7%. This matters because smaller images move faster, expose less unnecessary software, and reduce the amount of material that must be trusted in production. The service also runs as a dedicated non-root user, which reduces the damage that a container-level issue could cause.

The second improvement is traceability. The deployed version is no longer described only by a general version name. It is tied to the application version and the exact source revision used to build it. That gives the team a clearer answer when someone asks what is running, what changed, and what version should be restored if a problem appears. This is especially important for payments because rollback decisions need evidence, not memory.

| Concern              | How Week 7 addressed it                                                                                                 | How Week 8 addresses it                                                                                                                  |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| Deployment mechanism | Blue and green application versions were prepared on servers, then traffic was switched between them through the proxy. | A versioned container image is built, pushed to a private registry, pulled by the cluster, and started through a Kubernetes Deployment.  |
| Rollback mechanism   | Rollback returned traffic to the previous environment using the stored active and previous state.                       | Rollback can use the previous image tag and redeploy the known working version through the same deployment mechanism.                    |
| Failure recovery     | Recovery depended mainly on the rollback automation and the server process layout created by the scripts.               | The cluster automatically replaces a failed payments Pod and restores the desired replica count without manually restarting the service. |
| Scaling              | Scaling required preparing more server capacity or changing how the service was installed and routed.                   | Scaling is expressed as replica count, allowing the same packaged service to run as more than one instance behind one stable Service.    |

The self-healing evidence is the most practical difference for operations. In the fresh Week 8 deployment, one payments instance was deleted and the cluster restored service capacity in 68 seconds. That does not mean users would never notice a problem, but it does show that the system now has an automatic repair loop. Instead of relying only on someone noticing that a process disappeared, the platform compares the desired state with the actual state and starts a replacement.

This is also better for board-level confidence because the evidence is connected from end to end. The image was rebuilt from a clean state, stored in the private registry, pulled back after local deletion, deployed with two replicas, exposed through a stable service address, checked from outside the cluster, checked from inside the cluster, and then tested for replacement after failure. That sequence is stronger than showing separate lab steps because it proves the artefacts work together.

Week 8 does not yet make this production-complete. Configuration is still too closely tied to the deployment definition, and sensitive or environment-specific values need a cleaner separation before this can safely support multiple environments. The current health check also proves that the application can respond, but it does not fully control when traffic should begin flowing to a newly started instance. Week 9 adds configuration and secret management, which will make the deployment safer across development, staging, and production while reducing the risk of changing application definitions just to change environment settings.
